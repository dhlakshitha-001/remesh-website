Research Domain

Literature, Problem & Methodology

Theoretical foundation, identified research gaps, problem statement, objectives, methodology, and technologies underpinning REMESH.

Literature Survey
IoT-Based Flood Monitoring Systems

Basha & Rus (2007) established that wireless sensor networks at river monitoring points deliver significantly earlier warnings than centralised systems. Keoduangsine & Goodwin (2012) demonstrated that GPRS/internet-dependent systems lose connectivity precisely during active flood events — the fundamental failure mode REMESH is designed to overcome. Rizal et al. (2024) showed multi-sensor IoT designs integrating water level, flow, rainfall, temperature, and humidity improve flood status classification significantly over single-sensor approaches, validating the seven-sensor main node design.

LoRa Communication for Rural IoT

LoRa modulation achieves kilometre-scale transmission range at very low power, making it ideal for rural river basin sensor networks. Petäjäjärvi et al. (2015) confirmed 15 km suburban range at 868 MHz; 433 MHz provides even more favourable propagation characteristics. Dias & Grilo (2018) demonstrated that randomised forwarding delays in LoRa mesh networks reduce packet collision probability by over 60% compared to fixed-delay forwarding — the direct basis for REMESH's 1.5–6 s repeater backoff strategy.

Machine Learning for Flood Prediction

XGBoost (Chen & Guestrin, 2016), Random Forest (Breiman, 2001), and Gradient Boosting (Friedman, 2001) have demonstrated strong performance on environmental datasets. Tehrany et al. (2014) established Logistic Regression as competitive with ensemble methods where a dominant hydrological feature — such as dam water level — forms a near-linear decision boundary, explaining REMESH's finding of superior Logistic Regression recall. Chawla et al. (2002) introduced SMOTE for principled minority-class interpolative oversampling, essential given the 1.8% flood prevalence in the Deduru Oya dataset (78 flood days in 4,383).

Alert Systems & Participatory Sensing

SMS alerting remains the most reliable last-mile communication channel in developing nations with limited smartphone penetration (ITU, 2023). Goodchild (2007) established participatory citizen emergency reporting via spatial heat maps as a critical complement to fixed-sensor monitoring — enabling rapid identification of impact zones that fall between static deployment points. This paradigm is integrated into the REMESH dashboard as a geo-tagged citizen alert module.

Research Gap

Identified Gap

No prior system in literature integrates all four of the following into a single validated platform: (1) a self-healing LoRa mesh communication layer that operates without internet connectivity; (2) multi-sensor flood sensing (7 parameters) with automatic WiFi-to-LoRa fallback per transmission cycle; (3) embedded machine learning inference running on commodity ESP32 hardware (no cloud dependency); and (4) multi-channel community alerting (SMS + on-node audio + dashboard) — all validated against real hydrological data from a named Sri Lankan river basin. Existing systems address subsets of these requirements but invariably assume internet or cloud availability, making them unreliable in the very conditions that define a major flood event.

Research Problem

Problem Statement

Centralised flood monitoring systems in rural Sri Lanka depend on continuous internet connectivity and cloud infrastructure for data transmission, ML inference, and alert dissemination. During major flood events — when these systems are most critically needed — both internet connectivity and power infrastructure are disrupted, rendering existing deployments non-functional at the moment of peak community risk. There is a need for a low-cost, fully offline-capable, self-healing flood early warning system that operates end-to-end without internet dependency, specifically validated for tropical river basin conditions in Sri Lanka.

Research Objectives
  • O1 Design and implement a three-tier distributed IoT node architecture — main sensor nodes (7 sensors, GPS, solar), repeater nodes (LoRa relay, offline dashboard), and end nodes (WiFi bridge, audio alert) — with automatic WiFi-to-LoRa fallback and watertight solar-powered enclosures for deployment at Deduru Oya basin upstream monitoring points.
  • O2 Develop a custom binary packet compression scheme reducing LoRa payload by ~88% (600 bytes JSON → ~70 bytes binary), a lightweight XOR rolling-counter encryption scheme for resource-constrained ESP32 nodes, and a multi-hop mesh routing protocol with loop prevention, duplicate suppression, and collision-avoidance backoff.
  • O3 Construct a real 10-year hydrological dataset from the Moragaswewa gauging station (2015–2025), apply SMOTE oversampling to address 1.8% flood class prevalence, and perform comparative evaluation of five ML classifiers (Logistic Regression, SVM, XGBoost, Random Forest, Gradient Boosting) for binary flood risk prediction, selecting the best model for ESP32 deployment.
  • O4 Build a centralised Laravel web dashboard integrating real-time sensor monitoring with Rivernet.lk data, ML flood risk visualisation (Low/Moderate/High/Critical), SMS alerting, DFPlayer audio alert triggering, geo-tagged citizen emergency reporting with heat-map display, and role-based access control for public users, district operators, and system administrators.
Hardware Design — Main Node Sensor Array

ESP32 microcontroller at the centre, interfacing all 7 sensors plus LoRa, GPS, and battery monitor across Digital, ADC, UART, SPI, and I²C buses.

Methodology
⚙️

Phase 1 — Hardware Design & Assembly

ESP32 + SX1278 LoRa nodes assembled on veroboard with 3×18650 battery packs regulated by LM2596 buck converters and recharged by 5V 100 mA solar panels. Main node integrates all 7 sensors in a watertight enclosure with GPS antenna and INA219 battery monitor. Repeater and end nodes in sealed outdoor enclosures. Adaptive transmission interval (60 s at full charge → 1800 s at <20% battery) conserves energy during monsoon cloud cover.

📡

Phase 2 — LoRa Mesh Firmware

Custom SensorPacket struct encodes all telemetry as fixed-size binary fields (lat, lng, speed, sats, temp, hum, flow, rain×2, turb, pH, timestamp, hop trail) and copies directly into a uint8_t buffer via memcpy — reducing ~600 byte JSON to ~70 bytes. XOR rolling-counter encryption: C[i] = P[i] ⊕ K ⊕ n, where K = 0x5A and n is the per-packet counter prepended as frame byte 0. Repeater routing pipeline: (1) loop prevention via hop-trail DEVICE_ID check, (2) duplicate suppression via 15-entry timestamp circular cache, (3) hop augmentation appending {DEVICE_ID, RSSI}, (4) randomised 1.5–6 s re-encrypt and forward.

🤖

Phase 3 — ML Dataset Construction & Model Training

Primary data: Moragaswewa gauging station (7°43'39"N, 80°05'46"E), 321 verified days (2015–2024) + 6 days hourly from November 2025 Cyclone Ditwah (peak WL = 9.15 m, peak Q = 895.4 m³/s). Gap-filled via log-polynomial regression on Chilaw downstream gauge (R² = 0.663 for WL, R² = 0.724 for Q, 4,290-day record). Basin rainfall from 7 government rain gauges; meteorological data from Open-Meteo API across 21 lowland stations. Final dataset: 4,383 daily records, 11 features. Flood label: dam_water_level > 5.0 m → 78 flood days (1.8%). SMOTE generated 4,227 synthetic flood samples → balanced 50/50 training set of 8,610 rows. Five classifiers under 75/25 stratified split with 5-fold GridSearchCV tuning.

📊

Phase 4 — Dashboard Development

Laravel REST API exposing /api/v1/telemetry for end-node telemetry (full hop trail), /api/v1/traditional-telemetry for WiFi-direct main node data, and /api/v1/emergency-alerts/active polled every 60 s by end nodes to trigger DFPlayer audio. Integrates Rivernet.lk live river levels. Citizen emergency geo-tagged submissions aggregated as heat map. Role-based access: public / district operator / system admin.

🧪

Phase 5 — Lab Testing & Validation

LoRa open-field range test: 1 km confirmed at 14 dBm TX power (Ra-02 SX1278, 433 MHz); >10 km estimated for Deduru Oya basin flatlands based on datasheet specs and Petäjäjärvi et al. (2015). Sensor calibration against reference instruments. ML tested on original unaugmented 4,383-row dataset (not SMOTE-augmented), reflecting real-world class distribution. End-to-end latency: <3 s via WiFi/HTTP; <15 s via 2-hop LoRa mesh (dominated by randomised repeater backoff). Logistic Regression selected: FN = 3, highest recall (99.42%), lowest ESP32 compute overhead.

ML Model Comparison — Recall & False Negatives

Five classifiers evaluated on the original unaugmented test set (n = 1,096 records, 25% of 4,383). Lower false negatives (FN) is operationally critical — a missed flood event carries the greatest consequence.

Technologies Used
⚙️
ESP32
Dual-core 240 MHz with built-in WiFi/BT, 12-bit ADC, UART, SPI, I²C — main compute on all three node tiers
📡
LoRa SX1278 (Ra-02)
433 MHz, 14 dBm TX, −148 dBm sensitivity, ~10 km open terrain range · SPI interface
🐘
Laravel
PHP framework for REST API, dashboard, SMS dispatch, citizen reports, and role-based access control
🤖
scikit-learn + SMOTE
ML pipeline: StandardScaler, 5-fold GridSearchCV, SMOTE oversampling, 5 classifiers evaluated
☀️
Solar + 18650 LiPo
5V 100 mA solar panels + 3×18650 cells regulated by LM2596 buck converter for off-grid operation
🌊
Rivernet.lk Integration
Live official river water level data for the Deduru Oya basin supplementing IoT node readings
🔊
DFPlayer Mini
MP3 module on end nodes driving 4 Ω 5 W speaker for community-level audio flood alerts
📍
NEO-6M GPS
Ublox GPS module for main node geolocation via TinyGPS++ library · enables spatial dashboard mapping
🔒
XOR Rolling-Counter Encryption
Lightweight C[i]=P[i]⊕K⊕n scheme — semantically secure against replay, minimal CPU overhead on ESP32
